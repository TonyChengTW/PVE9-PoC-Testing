# 260515-LACP-HA-測試計劃

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 更新 LACP HA 與故障轉移測試規格，使其正確反映 Corosync 雙 ring 設計（ring0: 172.19.0.x, ring1: 10.23.0.x）的備援機制，並新增真正測試叢集 HA 觸發的條件。

**Architecture:**
- 測試分為兩大部分：TC-HA-02（HA 觸發驗證）與 TC-NW-02（故障轉移時間驗證）
- 環境有獨立 nic0 (172.23.0.x) 用於 Debug/Agent，不參與 Cluster LACP 測試
- Corosync 使用雙 ring，ring1 (10.23.0.x) 為 ring0 (172.19.0.x) 的備援路徑
- 真正測試 HA 觸發需同步阻斷 ring0 + ring1

**Tech Stack:** PVE Cluster, Corosync, LACP Bond, iptables, iperf3, ping

---

## 背景說明

### 原始 Spec 問題

原 TC-HA-02 假設：
- bond0 雙鏈路故障 → Corosync 失聯 → HA 觸發

**實際環境**：
- Corosync 已有 ring1 (10.23.0.x) 備援
- bond0 雙鏈路故障不影響叢集穩定性（Quorate 維持）
- 需阻斷**兩條 ring** 才能真正測試 HA

原 TC-NW-02 問題：
- 未考慮 Corosync ring1 的備援影響
- 故障轉移測試需同時驗證頻寬與 HA 穩定性

### 6.5 執行測試 — 雙 Ring 同步中斷（真正 HA 測試）

```bash
# 同步阻斷 ring0 (172.19.0.x) 和 ring1 (10.23.0.x) 與 172.19.0.172 的通訊
# 此測試會隔離 sandbox-pve02，觸發真正的 HA 條件

# 記錄測試前狀態
ha-manager status > /tmp/ha_status_before_full
pvecm status > /tmp/pvecm_before_full

# 執行同步隔離
iptables -A INPUT -s 172.19.0.172 -j DROP
iptables -A INPUT -s 10.23.0.172 -j DROP
echo "雙 Ring 隔離開始: $(date)"

# 等待 15 秒（考慮 deadtime）
sleep 15

# 檢查 Corosync 成員狀態
corosync-cmapctl | grep members

# 檢查 quorum
pvecm status
# 預期：
# - Quorate: No 或叢集分割
# - 172.19.0.172 應從 members 中消失

# 檢查 HA 是否觸發
ha-manager status
# 預期：
# - Fence 機制觸發
# - VM 被標記為 starting/migrating/fenced

# 恢復網路
iptables -D INPUT -s 172.19.0.172 -j DROP
iptables -D INPUT -s 10.23.0.172 -j DROP
pvecm status
# 預期：Quorate: Yes，三節點恢復
```
```
